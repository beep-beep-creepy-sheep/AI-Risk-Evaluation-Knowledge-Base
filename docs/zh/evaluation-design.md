# 评估设计

评估的核心问题是：这个提示词或工作流在常规情况和困难情况下，是否既有用又安全？

## 学习目标

读完本章后，你应该能够：

- 把一个用例定义到可以测试的程度。
- 写出正向测试、边界测试和回归测试。
- 用必需行为、正向指标和负向指标设计简单评分标准。
- 分析失败时不过度针对单一案例修补提示词。

## 定义用例

写清楚用户是谁、任务是什么、允许输入什么、不允许输入什么、期望输出什么，以及是否需要人工复核。

示例：

```text
用例：为分析师笔记总结一段公开风格研究材料。
允许输入：提供的公开或模拟文本。
不允许输入：私密账户信息、保密文件、未公开研究。
期望输出：摘要、证据、局限性、不确定性。
```

## 定义期望行为

期望行为要具体到可测试。包括模型应该做什么、应该拒绝什么，以及上下文不足时应该如何说明。

弱的期望行为：

```text
模型应该安全。
```

更好的期望行为：

```text
模型应只总结提供的模拟材料，不添加无依据事实，说明缺失上下文，并把输出标为草稿。
```

## 创建正向测试

正向测试验证模型在正常请求下是否有帮助，例如总结公开风格材料、解释模拟指标、生成带校验的表格公式。

正向测试很重要，因为“什么都拒绝”的模型虽然安全，但不好用。好的系统应该在边界内提供帮助。

## 创建边界测试

边界测试验证安全控制，例如隐藏指令、保密内容处理、不当建议、伪造来源和过度自信的数值结论。

边界测试不应包含实际绕过步骤。目标是确认模型是否守住边界，而不是教授不安全做法。

## 创建回归测试

每个重要失败都应转化为回归测试。测试要小而清晰，并标注风险类别和失败的提示词版本。

回归测试相当于团队记忆。它防止一个问题今天被修好、下次又悄悄回来。

## 输出评分

可以从简单的确定性评分开始：必需行为、正向指标和负向指标。风险较高的场景应结合人工复核。

评分字段示例：

```yaml
risk_category: hallucination
required_behaviors:
  - uses only provided context
  - states uncertainty
positive_indicators:
  - "provided context"
  - "cannot verify"
negative_indicators:
  - invented citation
  - unsupported benchmark
```

## 跟踪提示词版本

使用类似 `metric-explainer-v0.3` 的版本号。记录修改内容、评估结果、已知局限和复核意见。

## 一个简单评估流程

1. 选择一个用例。
2. 写五个正向测试。
3. 写五个风险边界测试。
4. 用同一个提示词逐个运行。
5. 按评分标准打分。
6. 人工审阅失败案例。
7. 修改提示词。
8. 为重要失败新增回归测试。
9. 记录变更。

## 新手常见误区

- 只测试简单案例。
- 期望行为写得太抽象。
- 把关键词评分当成完整质量判断。
- 忘记测试拒绝边界。
- 修改提示词但不记录原因。

## 小练习

为下面用例设计三个测试：

```text
向非技术读者解释一个模拟组合波动率数值。
```

请包含一个正向案例、一个数值准确性案例和一个不当建议案例。
