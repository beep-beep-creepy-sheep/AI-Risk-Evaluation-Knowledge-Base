# 评估设计

评估的核心问题是：这个提示词或工作流在常规情况和困难情况下，是否既有用又安全？

## 定义用例

写清楚用户是谁、任务是什么、允许输入什么、不允许输入什么、期望输出什么，以及是否需要人工复核。

示例：

```text
用例：为分析师笔记总结一段公开风格研究材料。
允许输入：提供的公开或模拟文本。
不允许输入：私密账户信息、保密文件、未公开研究。
期望输出：摘要、证据、局限性、不确定性。
```

## 定义期望行为

期望行为要具体到可测试。包括模型应该做什么、应该拒绝什么，以及上下文不足时应该如何说明。

## 创建正向测试

正向测试验证模型在正常请求下是否有帮助，例如总结公开风格材料、解释模拟指标、生成带校验的表格公式。

## 创建边界测试

边界测试验证安全控制，例如隐藏指令、保密内容处理、不当建议、伪造来源和过度自信的数值结论。

## 创建回归测试

每个重要失败都应转化为回归测试。测试要小而清晰，并标注风险类别和失败的提示词版本。

## 输出评分

可以从简单的确定性评分开始：必需行为、正向指标和负向指标。风险较高的场景应结合人工复核。

## 跟踪提示词版本

使用类似 `metric-explainer-v0.3` 的版本号。记录修改内容、评估结果、已知局限和复核意见。
