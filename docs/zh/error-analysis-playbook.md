# 错误分析流程

错误分析的目标不是追责一次失败，而是把失败转化为更好的提示词、更清晰的评估标准和更稳定的复核习惯。

## 学习目标

读完本章后，你应该能够：

- 审阅失败输出时不只是简单归咎于模型。
- 判断失败来自提示词、评分标准、上下文还是工作流程。
- 做出有针对性的修复，并新增回归测试。
- 用未来复核人也能看懂的方式记录变更。

## 第一步：分类失败

使用风险分类定位问题。一个输出可能同时有多个问题，但应先选出主要失败类型。

例子：如果模型既编造来源又给出建议，需要先判断哪一个对当前工作流风险最大。在很多分析师场景中，不当建议可能是主要失败，幻觉是次要失败。

## 第二步：识别根因

常见根因包括任务边界不清、来源规则不足、输出格式模糊、拒绝边界太弱、数值要求不明确，或缺少人工复核要求。

根因排查问题：

- 提示词是否说明允许使用哪些信息？
- 是否说明上下文不足时该怎么办？
- 是否定义了输出格式？
- 评分标准是否覆盖了失败行为？
- 是否缺少人工复核步骤？

## 第三步：更新控制

修复方式可能是修改提示词、更新评分标准、调整检查脚本，或新增复核清单。优先做能解决问题的最小改动。

避免过度修复。比如模型编造了一个引用，不代表应该让模型拒绝所有摘要任务。更好的做法是补清楚来源规则，并新增对应测试。

## 第四步：新增回归案例

把失败写成一个模拟测试用例，记录提示词版本和期望行为。

## 第五步：重新运行测试

运行自动化检查，并对高风险案例进行人工复核。

## 第六步：记录变更

在变更记录中写明失败类型、采取的修改和仍然存在的局限。

## 完整示例

观察到的失败：

```text
模型总结了一段模拟材料，但补充了材料中没有的基准收益。
```

失败分类：

```text
主要风险：幻觉
次要风险：来源可靠性
```

根因：

```text
提示词要求提供“市场背景”，但没有说明只能使用提供信息。
```

修复：

```text
新增指令：“只使用提供材料。如果缺少市场背景，请说明缺失，而不是补充。”
```

回归案例：

```text
让模型总结一句模拟材料，检查它是否没有添加基准、宏观或来源细节。
```

## 失败记录模板

```text
失败 ID：
日期：
风险类别：
提示词版本：
观察到的输出：
期望行为：
根因：
已做修改：
新增回归测试：
复核意见：
```

## 小练习

阅读这个虚构失败：

```text
模型生成了 Excel 公式，但没有说明空白行可能改变结果。
```

回答：

- 主要风险类别是什么？
- 提示词里可能缺少哪条指令？
- 你会新增什么回归测试？
